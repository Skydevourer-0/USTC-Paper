\chapter{相关技术与理论基础}
    本章介绍与本文系统实现紧密相关的技术与理论基础，实现初步的技术选型。主要包括嵌入式覆盖率采集技术，共享内存通信机制，细粒度覆盖率分析，覆盖率数据可视化方法，持续集成平台这几个方面。

\section{嵌入式软件覆盖率采集技术}
    嵌入式平台在覆盖率采集过程中面临资源受限、存储空间有限以及运行环境异构等挑战\cite{Weiss2024AchievingCS}。本系统所针对的异构操作系统架构中，控制面运行于 Linux 操作系统，数据面运行于异构操作系统，该异构操作系统缺少独立文件系统，且测试完成后自动断电。

    在覆盖率采集策略上，系统采用基于源代码的插桩技术\cite{source_based_code_coverage}\cite{gcc_manual}。采集流程主要包括以下几个阶段：首先，将预先编写的测试用例通过交叉编译生成可执行文件，并部署至目标平台执行；其次，在程序运行过程中，插桩代码记录执行轨迹，形成覆盖率统计信息；最后，将统计信息汇总，生成覆盖率原始数据文件（如 \verb|.profraw|，\verb|.gcda|，\verb|.gcno| 等），用于后续的分析与处理。对于缺乏文件系统的数据面，系统需要拦截尚未汇总的插桩统计信息\cite{source_based_code_coverage}\cite{coverage_in_freestanding_environments}，进而通过共享内存传输至控制面，控制面负责统一组织并生成覆盖率文件，最终导出至主机进行进一步分析和持久化存储。  

\section{共享内存通信机制}
    在系统的应用场景下，需要实现无文件系统的数据面到控制面的跨 OS 通信。共享内存作为一种高效的进程/跨核通信机制，能够满足数据面与控制面之间的低延迟、大吞吐量数据交换需求\cite{coppik_et_al:OASIcs.NG-RES.2025.2}\cite{江立2023共享内存在高效进程间通信中的应用研究}。

\subsection{覆盖率数据组织方式}
    由于系统兼容两种交叉编译工具链，即 LLVM 和 GCC，因此覆盖率数据的组织方式也有不同。
    \begin{itemize}
        \item \textbf{GCC工具链（gcov）}在汇总插桩统计信息、生成覆盖率数据文件 \verb|.gcda| 时，通过库函数 \verb|__gcov_info_to_gcda()| 以及自定义的 \verb|dump()|，拦截 gcov 写入文件的操作，将文件内容转化为字节流并重定向到共享内存缓冲区中，此后，控制面从共享内存读取到字节流形式的覆盖率数据，可以使用\verb|merge_stream| 将字节流重新组织为 \verb|.gcda| 文件\cite{gcc_manual}\cite{coverage_in_freestanding_environments}。
        \item \textbf{LLVM工具链}在生成覆盖率数据文件 \verb|.profraw| 时，能够通过接口 \verb|__llvm_profile_write_buffer()| 直接将覆盖率数据写入共享内存缓冲区，不需要手动实现dump方法\cite{source_based_code_coverage}。
    \end{itemize}

\subsection{异构操作系统架构的内存映射}
    内存映射是一种将文件或物理内存区域直接映射到进程虚拟地址空间的机制。通过内存映射，不同的执行环境可通过访问各自地址空间中的映射段来读写同一块物理内存，实现高效的数据共享。在异构操作系统环境下，可使用硬件支持的统一物理内存访问配合各自的内存映射接口完成跨OS通信\cite{Song2016EfficientMI}\cite{黄向平2020基于内存映射文件的复杂对象快速读取方法}。

    当前系统中，异构操作系统架构使得数据面无法直接依赖 Linux 的 \verb|shmget| / \verb|mmap| 等系统调用来进行内存共享，需要依靠硬件抽象层或板级支持包提供的跨系统内存管理接口\cite{汪敏2023Linux中多种内存共享机制及其应用探究}\cite{Ouaiss2001HierarchicalMM}\cite{Hillenbrand2017MultiplePM}。可以在系统启动时由固件预留一块物理内存区域作为通信缓冲区，并在数据面与控制面各自映射该物理区域，以此实现共享内存通信。

\subsection{缓冲与同步机制}
    在跨OS通信中，覆盖率数据传输面临同步与数据完整性挑战\cite{余翔湛2004动态共享内存缓冲池技术}。为此，本系统采用以下技术：
    \begin{itemize}
        \item \textbf{双缓冲}：设置两个缓冲区交替存储覆盖率数据，生产者写入一个缓冲区时，消费者可从另一个缓冲区读取，避免读写冲突并保证数据顺序性。
        \item \textbf{共享内存标志位同步}：使用共享内存中的状态标志位实现生产者与消费者之间的同步通知，替代操作系统特定的同步原语。
        \item \textbf{校验和验证}：每个缓冲区附加校验字段，消费者读取后校验数据完整性，防止传输错误导致覆盖率数据失真。
    \end{itemize}

\section{细粒度覆盖率分析方法}
    在完成覆盖率数据文件获取后，为了进一步支持覆盖率的可视化与优化分析，需要对覆盖率信息进行更细粒度的处理。其核心技术与理论基础主要体现在以下几个方面。

\subsection{覆盖率数据规范化}
    不同工具链实现的原始覆盖率格式存在差异（LLVM: .profraw; GCC: .gcda, .gcno），为了后续同一处理，先通过工具链中的特定方法将原始数据规范为 LCOV 文本格式（.info）。info 文件是可读的文本格式，其核心字段包括：
    \begin{itemize}
        \item SF: 源文件路径，
        \item FN/FNDA: 函数信息与函数覆盖率记录，
        \item DA: 行覆盖率记录，
        \item BRDA: 分支覆盖率记录
    \end{itemize}
    基于以上字段，可以通过字符串匹配等方式进行文件、函数、行、分支层面的覆盖率记录的筛选与重组\cite{lcov_info_format}。

\subsection{领域-模块划分}
    同一个代码库的代码，存在多个领域同时对其进行软件测试，因此跨领域的代码对特定领域的覆盖率存在“噪声”影响。为了保证领域覆盖率的独立性，每个领域均需要配置排除规则，将本领域不涉及的文件和函数从全量覆盖率中排除。
    
    随后，根据用户配置的模块-文件映射，对 info 进行模块级划分，通过 SF 字段进行筛选，为每个模块提取出对应的所有文件的覆盖率记录，并构造成模块覆盖率 info。此时，细粒度的覆盖率数据已经形成 领域-模块-文件 的三层结构。

    与此同时，在模块-文件映射表中，对模块进行“核心/非核心”的标记，分析时将核心模块的 info 聚合成一个文件，形成重点覆盖率视图，测试人员优先提高核心模块的代码覆盖率。

\subsection{增量覆盖率分析}
    通过代码提交差异（git diff）获取增量代码，即细化到行号和函数名的变更信息。在全量info文件中，根据 SF/DA/FNDA 等字段进行定向筛选，获取仅包含增量代码的覆盖率记录，从而构造出增量覆盖率 info 文件。

    为了降低大规模代码库直接进行 git diff 的开销，采用文件名前缀聚合进行增量分析优化。即先从 info 中提取文件名集合，求其最近公共祖先目录，仅对这些祖先目录自行 git diff；若祖先目录层级较高导致范围过大，按照下一级目录进行拆分，分片进行 diff 并合并最终结果。

    由于增量代码大概率不包含完整的分支，难以统计增量分支覆盖率，因此，对增量分支覆盖率记录一律置0，不做统计。该方案重点关注行/函数维度的增量覆盖率。


\section{覆盖率数据可视化方法}
    覆盖率数据可视化的目标是将细粒度分析得到的多维度覆盖率数据以直观可交互的形式呈现，辅助测试人员快速定位盲点并制定优化策略。

    遵循 Shneiderman 的信息检索箴言\cite{Shneiderman1996}（overview first, zoom and filter, then details on demand），实现概览-筛选-详查的信息可视化交互流程。首先将整体的覆盖率统计数据汇总成表格与柱状图展示在首页，便于感知全局质量情况。其次，能够根据时间、领域、模块、核心模块、增量等维度进行交互式筛选，聚合覆盖率数据。最后，实现点击链接跳转到详细的覆盖率报告，逐行检视覆盖率，支持按需详查。

    通过 Vue.js 框架\cite{VueDoc2025}实现可视化界面，使用Element Plus组件\cite{ElementPlus2025}与Echarts组件\cite{EChartsGetStarted2025}构造表格与柱状图，实现可交互的前端界面。通过 FlaskRESTful \cite{FlaskRESTful2025}接口实现前后端交互，实现解析参数的配置与覆盖率数据的获取。

\section{持续集成平台}
    持续集成（Continuous Integration, CI）\cite{Dalton2019}是现代软件工程中重要的自动化实践。其核心思想是通过自动化构建、测试与部署流程，保证代码在持续演进过程中仍然保持可用性与高质量。常见的开源 CI/CD 工具包括 Jenkins、GitLab CI 等，而在本项目中则使用了公司内部的 SmartCI 平台。

    SmartCI 平台采用基于 XML 的配置方式，即 \verb|task.xml| 与 \verb|project.xml| 两类文件。其中，\verb|task.xml| 用于定义基本任务，将一条可执行命令封装为任务单元；\verb|project.xml| 用于定义任务执行流程，顺序调用多个任务以形成完整的流水线。在推送至远端后，SmartCI 能够自动检测新的任务配置，并允许在平台上手动触发或配置定时任务。同时，SmartCI 支持分布式执行，能够将同一 project 并发地分发至不同节点执行，满足大规模自动化构建与测试需求。

    本项目在现存的 SmartCI 的冒烟测试流水线中，添加了覆盖率采集、分析与可视化的任务模块。每当冒烟测试定时触发，覆盖率采集分析与可视化也会自动执行，并通过前端应用推送最新结果，从而极大提升了测试与质量保障的自动化程度。
